## 1. Introduction

The **Face Recognition-Based Attendance Management System** is a Flask-based web application that automates student attendance using real-time face recognition. It combines role-based access control (Admin, Faculty, Student), webcam-based face capture, facial encoding, live recognition, and attendance reporting in one integrated platform.

In real-world terms, it solves the operational problems of manual attendance: slow roll-calls, proxy attendance, delayed report generation, and fragmented record-keeping. The system allows faculty to run attendance sessions with a camera feed while the backend verifies faces and records attendance directly into a centralized database.

---

## 2. Problem Statement

Traditional attendance systems face multiple issues:

- Manual methods are time-consuming and reduce teaching time.
- Proxy attendance is common in paper/signature-based systems.
- Attendance records are often distributed across sheets/files, making audits difficult.
- Generating daily/monthly attendance reports requires additional manual effort.
- Existing digital systems without biometric verification cannot ensure physical presence.
- Access control is often weak; users may view/modify data beyond their role.

---

## 3. Objectives of the Project

- Automate attendance marking using live face recognition.
- Reduce manual workload for faculty and administrators.
- Prevent duplicate attendance marking for the same student/course/day.
- Improve attendance accuracy through identity verification.
- Provide role-based dashboards for Admin, Faculty, and Student.
- Support student onboarding from account creation to face registration.
- Generate downloadable daily and monthly CSV reports.
- Maintain compatibility with older database schema through automatic migration checks.

---

## 4. Literature Review

Current attendance solutions in this domain generally include:

- **Manual registers / spreadsheets**: low cost, but error-prone and not scalable.
- **RFID / smart-card systems**: faster than manual methods but vulnerable to card sharing.
- **Fingerprint biometrics**: good verification but requires touch hardware and maintenance.
- **Face recognition systems**: contactless and user-friendly, often using:
  - Face detection (Haar/HOG/CNN-based)
  - Face embedding generation (e.g., 128-D vectors)
  - Distance-based matching (Euclidean thresholding)

Common limitations in existing solutions:

- High-end systems require expensive infrastructure.
- Many prototypes lack role-based workflow and reporting integration.
- Recognition-only demos often omit enrollment/course mapping/business logic.
- Some solutions do not handle operational concerns like duplicate prevention and export-ready reporting.

## 5. Proposed Methodology

The proposed system uses a practical pipeline from enrollment to attendance logging:

1. **User and academic setup**
- Admin creates faculty/student accounts.
- Admin creates courses and maps faculty.
- Admin enrolls students into courses.

2. **Face registration**
- Faculty opens registration module.
- Webcam captures multiple student images (`capture_faces`).
- System processes images and generates facial encodings (`generate_encodings_for_user`).
- Encodings are synced to a serialized cache (`encodings.pkl`) for fast recognition.

3. **Live attendance session**
- Faculty starts attendance for a selected course.
- Recognition runs in a separate thread (`start_attendance` route + `threading.Thread`).
- Webcam stream is processed frame-by-frame (optimized by processing alternate frames).
- Faces are detected (HOG), encoded, and compared with known encodings.
- Best match is selected using minimum face distance and threshold checks.

4. **Attendance logging**
- If confidence threshold passes, attendance is inserted into DB.
- DB-level unique constraint blocks duplicates (`UNIQUE(name, date, course_id)`).

5. **Report generation**
- Admin/Faculty views filtered reports in web UI.
- Daily/monthly CSV exports are generated by attendance manager.

Key techniques used:

- HOG-based face location for CPU-friendly inference.
- Euclidean distance-based face matching from `face_recognition`.
- Average encoding per user during registration for robustness.
- Threaded recognition to avoid blocking Flask request handling.
- SQLite relational schema for users, courses, enrollments, and attendance.

## 10. Algorithms Used

1. **HOG-Based Face Detection**
- Used in: registration and live recognition pipeline (`face_recognition.face_locations(..., model="hog")`).
- Logic: extracts gradient orientation features to localize faces in images.
- Why chosen: reliable CPU-based detection without GPU dependency.

2. **Face Embedding Extraction (128-D vectors)**
- Used in: `face_recognition.face_encodings`.
- Logic: transforms each detected face into a compact numeric feature vector.
- Why chosen: robust identity representation across moderate pose/lighting variation.

3. **Euclidean Distance Face Matching**
- Used in: `face_recognition.face_distance` + `compare_faces`.
- Logic: computes distance between live embedding and known embeddings; minimum distance indicates best candidate.
- Why chosen: fast, interpretable, and standard in practical face-recognition systems.

4. **Average Encoding Aggregation**
- Used in: registration (`np.mean(user_encodings, axis=0)`).
- Logic: multiple images per user are averaged to reduce single-image noise.
- Why chosen: improves stability over one-shot enrollment.

5. **Frame Skipping Optimization**
- Used in: detector loop (`process_this_frame` toggle).
- Logic: process every alternate frame to reduce compute load.
- Why chosen: improves responsiveness on normal lab laptops.

## 12. Project Workflow (Step-by-Step Execution)

1. Start server using `python app.py`.
2. User opens web app and logs in.
3. System verifies credentials and role.
4. Admin sets up faculty, students, courses, and enrollments.
5. Faculty opens registration page and selects/enters student.
6. Capture module acquires face images from webcam.
7. Encoder generates student embeddings and updates encoding store.
8. Faculty starts attendance for a selected course.
9. Flask spawns recognition thread; camera window opens.
10. Detector processes live frames, matches faces, and computes confidence.
11. For valid matches, attendance manager requests DB insertion.
12. Database records attendance or rejects duplicate same-day same-course entries.
13. Faculty/admin views attendance in reports page.
14. User exports daily/monthly CSV files if required.
15. Student logs in to view personal attendance history.

## 18. Limitations

- Recognition quality depends on lighting, camera angle, and image quality.
- No password hashing in current implementation (demo-oriented security tradeoff).
- Single-camera local deployment; not designed for distributed multi-camera setups.
- Potential false positives/negatives under occlusion or extreme pose changes.
- Webcam contention can occur if another app is using the camera.
- Current approach is largely CPU-bound; high throughput scenarios may need GPU/CNN pipeline.
- No anti-spoofing (photo/video replay attack protection) in current version.

## 19. Troubleshooting / Edge Cases

- **Camera not opening**
  - Cause: device busy (Zoom/Meet/other process).
  - Handling: retry logic in capture/detector; Linux lock cleanup command for `/dev/video0`.

- **No face encodings found**
  - Cause: student not registered or poor capture quality.
  - Handling: detector attempts `_sync_all_encodings`; prompts user to register first.

- **Registration captured but encoding failed**
  - Cause: no valid face detected in stored images.
  - Handling: registration flow reports failure and suggests better lighting/retry.

- **Attendance not marked despite recognition**
  - Cause: low confidence threshold or face mismatch.
  - Handling: marks only above threshold to avoid false positives.

- **Duplicate marks**
  - Cause: repeated recognition in same session/day.
  - Handling: blocked by unique DB constraint.

- **Thread/session conflicts**
  - Cause: starting a new attendance session while one is active.
  - Handling: route-level check prevents parallel recognition sessions.

- **Window closes unexpectedly**
  - Cause: OS/OpenCV event issues.
  - Handling: robust cleanup in `finally` block, repeated `waitKey` event pumping, explicit release calls.
